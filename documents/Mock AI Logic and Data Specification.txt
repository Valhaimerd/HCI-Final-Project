Below is **Document 5 of 7: Mock AI Logic and Data Specification** for Codex. This defines how the prototype should simulate AI behavior without building a real CNN model or connecting to an AI API. The project documents describe the AI as an image-based advisory system that returns possible rice pest/disease results, confidence, explanation, treatment, prevention, and human-control options.  

````markdown
# Document 5: Mock AI Logic and Data Specification for Codex

## Project

RiceCare AI  
Pest and Disease Advisory for Rice Farmers

## Purpose of This Document

This document explains how Codex should simulate the AI behavior of RiceCare AI.

The prototype must not use a real AI model, real CNN, real training dataset, real backend, or real API.

Instead, the app should use mock AI logic with predefined sample results. This allows the prototype to demonstrate the full Human-Centered AI flow while staying simple enough for GitHub and Netlify deployment.

---

# 1. Main Rule

Use mock AI only.

Do not create or connect:

- Real CNN model
- Real machine learning training pipeline
- Python AI backend
- AI API
- Database
- Authentication system
- Real technician messaging
- Real disease detection

The AI should be simulated through JavaScript data and simple front-end logic.

---

# 2. Mock AI Flow

The app should simulate this flow:

```text
User selects or simulates plant image
→ User clicks Analyze Image
→ App shows AI Analysis Loading screen
→ App selects one mock result
→ App displays AI Result screen
→ User can view recommendations, save result, ask technician, report wrong result, or retake photo
````

The mock AI should feel realistic, but it should not pretend to be a real diagnosis.

Always use advisory language.

Correct wording:

```text
Possible Issue
May match symptoms
Advisory result
Ask technician for confirmation
```

Avoid:

```text
Confirmed disease
Final diagnosis
Guaranteed result
Apply treatment now
```

# 6. Mock Analysis Utility

Create this file:

```text
src/utils/mockAnalysis.js
```

It should import the mock result data and return one result.

Suggested code:

```js
import { mockAiResults } from "../data/mockAiResults";

export function getMockAiResult(mode = "random") {
  if (mode !== "random") {
    const selected = mockAiResults.find((result) => result.id === mode);
    if (selected) return selected;
  }

  const randomIndex = Math.floor(Math.random() * mockAiResults.length);
  return mockAiResults[randomIndex];
}
```

Codex may also support these mode aliases:

```js
const modeMap = {
  "high-risk": "rice-blast",
  "medium-risk": "brown-spot",
  "pest": "pest-damage",
  "healthy": "healthy-plant",
  "uncertain": "uncertain"
};
```

This allows easy testing of different result screens.

# 7. Confidence and Risk Rules

Use these rules to display confidence and risk.

```text
85–100% confidence:
High confidence

70–84% confidence:
Moderate confidence

50–69% confidence:
Low confidence

Below 50% confidence:
Uncertain result / Needs review
```

Risk levels:

```text
Low
Medium
High
Needs Review
```

Suggested display behavior:

```text
High Risk:
Show warning card and recommend technician confirmation.

Medium Risk:
Show caution card and recommend monitoring or technician help if symptoms spread.

Low Risk:
Show normal care tips and continued monitoring.

Needs Review:
Do not force a diagnosis. Recommend retaking photo or asking a technician.
```

# 18. Safe Advisory Logic

Codex must enforce safe advisory wording.

When risk is high:

```text
Show technician advice clearly.
Warn before chemical treatment.
Do not make treatment sound mandatory.
```

When confidence is low:

```text
Recommend retaking the photo.
Recommend technician review.
Do not show a strong disease claim.
```

When result is healthy:

```text
Encourage continued monitoring.
Do not say the plant is guaranteed healthy.
```

When recommending treatment:

```text
Use cautious wording.
Include technician confirmation for chemicals.
```

# 20. Final Rule

The mock AI should support the story of the prototype.

It must make the app feel functional, but it must never pretend to be a real deployed agricultural diagnosis system.

RiceCare AI is an advisory prototype. The farmer remains the final decision-maker, and human technician confirmation must always be available.
