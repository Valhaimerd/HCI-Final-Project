# Document 4: Expanded Screen and Flow Requirements for Codex

## Project

RiceCare AI  
Pest and Disease Advisory for Rice Farmers

## Purpose of This Document

This document defines the screens, states, and user flows that Codex must build for the RiceCare AI prototype.

The prototype must not be limited to only 15 screens. The uploaded UI Design document lists 15 required screens, but the actual app needs additional states, modals, and alternate flows to make the prototype complete and realistic.

Some items in this document should be full pages. Some may be modal dialogs, conditional states, or reusable screen variations.

The final prototype must feel like one complete mobile app, not disconnected screens.

---

# 1. Global Screen Rules

Every screen must follow the RiceCare AI design system.

Use:

- Mobile-first layout
- Consistent app identity
- Consistent header
- Consistent bottom navigation where appropriate
- Green agriculture-focused visual style
- Simple farmer-friendly wording
- Clear advisory language
- Large readable buttons
- Card-based layout
- Scrollable content when needed

Do not use technical AI terms on farmer-facing screens.

Use “possible issue,” “advisory result,” “confidence,” and “ask technician.”

Do not use “confirmed diagnosis,” “guaranteed result,” or “apply chemical now.”

---

# 8. Button Behavior Requirements

Every visible button must do something.

No dead buttons.

Minimum expected behavior:

```text
Primary action buttons navigate to the next screen.
Back buttons return to the previous logical screen.
Save buttons show confirmation.
Submit buttons show submitted state.
Delete buttons show confirmation first.
Bottom navigation changes active screen.
Settings toggles visibly change state.
```

If real functionality is not available, simulate it.

---

# 9. Form Behavior Requirements

Forms should:

```text
Accept typed input
Have clear labels
Use placeholder examples
Show submit action
Navigate to confirmation/submitted screen
```

Required forms:

```text
Technician Support Form
Report Wrong Result Form
Manage Consent Form
Settings Form
```

Optional form validation:

```text
Required name/location fields for technician support
Required explanation field for report wrong result
Disabled submit button until minimum fields are filled
```

Keep validation simple.

# 11. Accessibility Behavior Requirements

The Settings screen must include accessibility controls.

Minimum behavior:

```text
Large Text mode increases readable text size.
High Contrast mode improves text and border contrast.
Voice Guidance toggle changes on/off state.
Simple Mode toggle changes on/off state.
Language selector changes selected option.
```

These can be simulated locally.

The goal is to visibly demonstrate accessibility, not to fully translate the app.
