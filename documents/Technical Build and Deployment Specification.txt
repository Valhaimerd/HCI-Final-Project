# Document 2: Technical Build and Deployment Specification for Codex

## Project

RiceCare AI  
Pest and Disease Advisory for Rice Farmers

## Purpose of This Document

This document tells Codex how to build and deploy the RiceCare AI prototype.

The goal is to create a polished mobile-first web prototype that can be pushed to GitHub and deployed to Netlify.

This is a class prototype. Build it as a static front-end application with mock AI behavior.

Do not build a real AI model, backend, database, login system, or admin panel unless requested later.

---

# 1. Recommended Technical Stack

Use the following stack:

- React
- Vite
- JavaScript
- CSS Modules or plain CSS
- React Router
- Local mock data
- Local state using React hooks
- Netlify for deployment
- GitHub for source code hosting

Do not use TypeScript unless specifically requested.

Do not use a backend server.

Do not use Firebase, Supabase, MongoDB, Express, Flask, Django, Laravel, or any database.

Do not connect to a real AI API.

---

# 2. Application Type

Build RiceCare AI as a static mobile web prototype.

The prototype should work in a browser and look like a mobile application.

The app must be usable on:

- Desktop browser preview
- Mobile browser
- Netlify deployed URL

The design should prioritize a mobile portrait layout.

Recommended mobile frame target:

- Width: 390 px
- Height: 844 px or scrollable
- Mobile-first responsive layout

The app should still display properly on larger screens by centering the mobile app frame.

---

# 3. Deployment Target

Primary deployment target:

Netlify

Source control target:

GitHub repository

Optional compatibility:

GitHub Pages

The final output must produce a working deployed URL that can be placed inside the project PDF.

---

# 4. Important Technical Rule

The AI must be simulated.

Use mock AI logic only.

The app should simulate image analysis when the user clicks “Analyze Image.”

Possible behavior:

1. User clicks “Take Photo” or “Upload from Gallery.”
2. App displays a mock selected image preview.
3. User clicks “Analyze Image.”
4. App shows an AI loading screen.
5. After a short delay, app displays one mock result.
6. User can view recommendations, ask technician, report wrong result, save result, or retake photo.

The mock result can be selected from predefined sample data.

# 9. Image Upload Prototype Behavior

Do not implement real image analysis.

For file upload:

* Allow the user to select an image from device if possible.
* Show the selected image preview.
* If no real image is selected, use a plant placeholder image.
* Continue to the mock analysis flow.

For “Take Photo”:

* Simulate photo capture.
* Show a placeholder rice plant image.
* Continue to image preview or scan confirmation.

The prototype does not need actual camera access unless easy to add without breaking the build.

If camera access is implemented, provide fallback behavior when permission is denied.

# 10. Bottom Navigation Rules

Bottom navigation should appear on main app screens after consent.

Tabs:

* Home
* Scan
* History
* Help
* Settings

Routes:

```text
Home      → /home
Scan      → /scan
History   → /history
Help      → /help
Settings  → /settings
```

Active tab must be visually highlighted.

Bottom navigation should not appear on:

* Welcome screen
* Consent screen
* AI loading screen if it makes the screen cleaner
* Confirmation modal-only states if inappropriate

Use consistent behavior across the app.

---

# 11. UI Responsiveness

The app must be mobile-first.

On desktop:

* Center the mobile app frame.
* Use a soft background outside the app frame.
* Keep the app width around 390 px to 430 px.
* Allow vertical scrolling.

On mobile:

* Use full viewport width.
* Preserve safe spacing.
* Avoid horizontal scrolling.

All buttons must be large enough to tap.

Avoid cramped layouts.

---

# 12. Accessibility Requirements

Implement these prototype settings:

* Text Size: Normal / Large
* Language: English / Filipino / Local Language placeholder
* Voice Guidance: On / Off
* High Contrast Mode: On / Off
* Simple Mode: On / Off

These settings can be simulated visually.

High Contrast Mode should increase contrast and make borders/text stronger.

Large Text should visibly increase body text and button text size.

Simple Mode can reduce extra helper text or simplify some sections if easy to implement.

Do not make accessibility settings decorative only. They should visibly affect the UI where reasonable.

# 14. Netlify Configuration

Create a file:

```text
netlify.toml
```

Use this configuration:

```toml
[build]
  command = "npm run build"
  publish = "dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

This is required so React Router routes work correctly after deployment.

# 19. Netlify Deployment Steps

Deployment process:

1. Push project to GitHub.
2. Open Netlify.
3. Choose “Add new site.”
4. Choose “Import an existing project.”
5. Connect the GitHub repository.
6. Set build command:

```bash
npm run build
```

7. Set publish directory:

```text
dist
```

8. Deploy the site.
9. Copy the deployed Netlify URL.
10. Test all routes and buttons.
11. Use the Netlify URL as the prototype link in the final PDF.
