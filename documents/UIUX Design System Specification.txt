# Document 3: UI/UX Design System Specification for Codex

## Project

RiceCare AI  
Pest and Disease Advisory for Rice Farmers

## Purpose of This Document

This document defines the required visual design system for the RiceCare AI prototype.

Codex must follow this design system consistently across the entire app.

The final output must look like one complete, polished mobile application, not separate random screens.

---

# 1. Design Goal

Create a clean, simple, farmer-friendly, accessible mobile web app prototype.

The app must feel:

- Practical
- Calm
- Trustworthy
- Agriculture-focused
- Easy to understand
- Safe for non-technical users

The app must not look like:

- A social media app
- A chatbot-only app
- A sci-fi AI system
- A complex analytics dashboard
- A medical diagnosis app
- A random collection of screens

The user should quickly understand that RiceCare AI helps farmers check possible rice plant pests and diseases, but the AI only gives advisory support.

---

# 2. App Identity

Use the same app identity across all screens.

## App Name

RiceCare AI

## Subtitle

Pest and Disease Advisory for Rice Farmers

## Main Message

AI can help, but human confirmation is available.

## Required Advisory Disclaimer

Use this disclaimer where appropriate:

“RiceCare AI provides advisory support only. It is not a final expert diagnosis.”

## Required Risk Awareness Note

Use this note where appropriate:

“Image quality, lighting, crop variety, and local field conditions may affect AI accuracy.”

---

# 3. Target Visual Style

The visual style should be:

- Mobile-first
- Clean
- Warm but not childish
- Simple but not plain
- Modern but not futuristic
- Agriculture-related
- Easy to read outdoors
- Comfortable for farmers with low digital literacy

Use green as the dominant color because it connects to agriculture, rice plants, safety, and growth.

Avoid overusing bright colors. Use warning colors only when needed.

---

# 4. Mobile Layout Rules

Use one consistent mobile frame.

Recommended frame:

```text
Width: 390 px
Height: 844 px
Orientation: Portrait
````

The app must support vertical scrolling.

On desktop preview:

* Center the mobile frame horizontally.
* Keep the app width between 390 px and 430 px.
* Use a soft neutral background outside the mobile frame.
* Do not stretch the app into a wide desktop layout.

On mobile browser:

* Use full viewport width.
* Avoid horizontal scrolling.
* Respect safe areas at the top and bottom.
* Keep buttons easy to tap.

---

# 5. Spacing System

Use an 8-point spacing system.

Recommended values:

```text
Extra small spacing: 4 px
Small spacing: 8 px
Medium spacing: 12 px
Default spacing: 16 px
Large spacing: 24 px
Extra large spacing: 32 px
```

Screen padding:

```text
Horizontal screen padding: 24 px
Vertical section spacing: 16 px to 24 px
Card internal padding: 16 px
Spacing between cards: 16 px
Spacing between related text elements: 8 px to 12 px
```

Avoid cramped screens.

For long content, use scrollable pages instead of shrinking everything into one view.

---

# 6. Color System

Use this exact color palette.

```css
:root {
  --primary-green: #2E7D32;
  --secondary-light-green: #A5D6A7;
  --soft-green-background: #F8FAF7;

  --accent-yellow: #FBC02D;
  --warning-orange: #F57C00;
  --error-red: #D32F2F;
  --success-green: #388E3C;

  --card-background: #FFFFFF;
  --main-background: #F8FAF7;

  --text-dark: #1B1B1B;
  --text-secondary: #5F6B5F;

  --border-light: #DDE8DD;
  --disabled-gray: #B8B8B8;
}
```

## Color Usage Rules

Primary Green `#2E7D32`

Use for:

* Primary buttons
* Active navigation tab
* Important icons
* Main brand elements
* Success emphasis when appropriate

Secondary Light Green `#A5D6A7`

Use for:

* Soft highlights
* Secondary backgrounds
* Light agricultural accent areas

Soft Green Background `#F8FAF7`

Use as:

* Main app background
* Page background
* Calm neutral base

Accent Yellow `#FBC02D`

Use for:

* Medium risk
* Important tips
* Mild caution badges

Warning Orange `#F57C00`

Use for:

* High risk warning
* Report wrong result
* Treatment caution
* Urgent but non-error actions

Error Red `#D32F2F`

Use only for:

* Delete result
* Serious warning
* Error state
* Failed action

Success Green `#388E3C`

Use for:

* Saved state
* Submitted state
* Completed review step
* Healthy result when appropriate

# 8. Typography

Use a clean readable sans-serif font.

Preferred fonts:

```text
Inter
Roboto
SF Pro
System UI fallback
```

Recommended CSS font stack:

```css
font-family: Inter, Roboto, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

## Typography Scale

```text
Large Title: 28 px, bold
Screen Title: 22 px, bold
Section Title: 18 px, semibold
Card Title: 16 px, semibold
Body Text: 14–16 px, regular
Small Helper Text: 12–13 px, regular
Button Text: 15–16 px, semibold
Bottom Nav Label: 11–12 px, medium
```

## Typography Rules

* Keep text readable on mobile.
* Avoid long paragraphs inside UI cards.
* Use short explanations.
* Avoid tiny gray text below 12 px.
* Use dark text for important information.
* Use secondary text only for helper text.
* Do not use too many font weights.
* Do not use decorative fonts.

Right side: optional icon button

Header rules:

* Keep the title clear.
* Use dark text.

# 17. Confidence Indicator

Use a progress bar or circular indicator to show confidence.

Preferred style:

```text
Horizontal confidence bar
Label: 86% Confidence
Bar background: #DDE8DD
Bar fill: #2E7D32 for normal/high confidence
Bar fill: #FBC02D for medium confidence
Bar fill: #F57C00 for low or uncertain confidence
```

Rules:

* Confidence must not imply certainty.
* Always pair confidence with advisory language.
* Low confidence must suggest retake photo or technician review.

Example text:

```text
86% Confidence
The image strongly matches known symptoms, but this is still advisory support only.

# 19. Recommendation Layout

Recommendations must be practical and easy to read.

Use separate cards:

```text
Treatment Steps
Prevention Tips
Recommended Farming Practice
Safety Warning
```

Use numbered steps or short bullet-like rows.

Do not create long paragraphs.

Example:

```text
Treatment Steps
1. Remove severely infected leaves when possible.
2. Avoid excessive nitrogen fertilizer.
3. Improve field drainage and airflow.
4. Consult a technician before applying fungicide.
```

Use a warning card for chemical treatment advice.

# 24. Accessibility Design Rules

The app must support accessibility visually and functionally.

Minimum rules:

* Use readable font sizes.
* Do not use tiny gray text for important content.
* Buttons must be large enough to tap.
* Color contrast must be strong enough.
* Icons should support text, not replace it.
* High contrast mode should visibly improve contrast.
* Large text mode should increase readable text.
* Simple mode should reduce clutter where possible.

Avoid relying on color alone.

Example:

Do not show only a red badge. Also write “High Risk.”

---

# 25. High Contrast Mode

When High Contrast Mode is active:

* Make text darker.
* Make borders stronger.
* Reduce pale low-contrast text.
* Increase button contrast.
* Keep backgrounds clean.

Suggested CSS class:

```css
.high-contrast {
  --text-dark: #000000;
  --text-secondary: #2F3A2F;
  --border-light: #8AA58A;
  --main-background: #FFFFFF;
  --card-background: #FFFFFF;
}
```

---

# 26. Large Text Mode

When Large Text Mode is active:

* Increase body text.
* Increase helper text slightly.
* Increase button text.
* Preserve layout spacing.

Suggested behavior:

```text
Body text: +2 px
Button text: +1–2 px
Small helper text: +1 px
```

Do not break the layout when text size increases.

---

# 27. Simple Mode

When Simple Mode is active:

* Reduce non-essential helper text.
* Keep primary actions visible.
* Prefer shorter explanations.
* Keep warnings and advisory notes visible.
* Do not remove safety information.

Simple Mode should make the app easier, not less safe.

---

# 28. Image and Illustration Style

Use simple placeholder illustrations if real images are not available.

Recommended visuals:

* Rice leaf icon
* Rice plant image placeholder
* Camera frame illustration
* Detected symptom markers
* Technician support icon
* Shield privacy icon

Image rules:

* Do not use random stock images with inconsistent style.
* Do not use dark or scary disease graphics.
* Use rounded image containers.
* Add labels when showing highlighted symptom areas.
* Use simple circles or outline boxes for detected areas.

---

# 29. Advisory Language Rules

The UI must be careful with AI claims.

Use:

```text
Possible Issue
May match symptoms
Advisory result
Ask technician
Human confirmation
Risk level
Confidence score
```

Avoid:

```text
Final diagnosis
Guaranteed result
The AI knows
Treatment required
Apply chemical now
Disease confirmed
```

The app must never pressure the farmer to apply chemical treatment without technician confirmation.

---

# 30. Page Length and Scrolling

Do not force all content into one screen.

Use scrollable layouts for:

* Result details
* Recommendations
* Help articles
* Settings
* History
* Consent details

Important buttons may be placed near the bottom, but make sure they are easy to reach.

For very long pages, group content into cards.

---

# 31. Animation and Interaction Style

Use small, calm interactions only.

Acceptable:

* Loading spinner
* Progress bar
* Gentle card fade-in
* Button hover/pressed state
* Step progress in analysis screen

Avoid:

* Excessive animations
* Flashing effects
* Fast motion
* Distracting AI effects
* Game-like transitions

This is a practical agricultural tool.

---

# 32. Required UI Consistency Rules

Codex must ensure:

* Same color palette across all screens
* Same button styles
* Same card style
* Same bottom navigation
* Same typography scale
* Same spacing system
* Same app tone
* Same advisory language
* Same mobile frame width
* No disconnected screen designs

The prototype must feel like one real mobile app.

---

# 33. Visual Quality Checklist

Before finishing, verify:

* The welcome screen clearly introduces RiceCare AI.
* The consent screen is readable and not intimidating.
* The home dashboard makes scanning the main action.
* The scan screen is simple and practical.
* The result screen shows explanation, confidence, risk, and user control.
* The recommendation screen uses short practical steps.
* The technician review flow clearly shows human confirmation.
* The history screen includes audit trail evidence.
* The help screens are easy to scan.
* The settings screen visibly supports accessibility and privacy.
* The bottom navigation works consistently.
* The mobile layout is not cramped.
* The app looks polished enough for a final class prototype.
